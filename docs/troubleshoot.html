Troubleshooting

Symptoms, causes, and fixes for the most common Burrow failures. Start with the log stream command — it narrows most issues within seconds.

Reading the system log

Burrow writes all diagnostic output to the macOS unified log under subsystem com.krzemienski.burrow. Stream live output:

log stream --predicate 'subsystem == "com.krzemienski.burrow"' --level debug

Log categories: tunnel, cloudflare, network, ui, keychain, lifecycle. Filter to tunnel events only:

log stream --predicate 'subsystem == "com.krzemienski.burrow" AND category == "tunnel"'

All --token values are scrubbed before reaching the log — you will see --token [REDACTED] in any command-line captures. Full DNS API response bodies are stripped in error paths; only the HTTP status and error message are logged.

Menu bar icon stays gray or shows red dot

Token invalid (401)

Symptom: red dot, Cloudflare tab shows "API token rejected." Cause: the token was revoked or expired at dash.cloudflare.com/profile/api-tokens. Fix: create a new token with the four required scopes (see Installation → Step 1), then open Settings → Cloudflare → Change token.

Token missing scope

Symptom: red dot, Settings shows a scope diff listing the missing permissions. Fix: edit the token at the Cloudflare dashboard and add the missing permission, or create a fresh token. Burrow re-validates automatically within 60 seconds of a Keychain write.

cloudflared not found

Symptom: gray icon at startup, notification "Install with brew install cloudflared." Cause: BinaryLocator searched /opt/homebrew/bin, /usr/local/bin, and /usr/bin and found nothing. Fix: run brew install cloudflared, then open Settings → Advanced and click Detect path. For non-Homebrew installs, paste the full path in the custom path field and Burrow validates it with cloudflared --version.

Tunnel starts then immediately exits

Run token stale

Cause: cloudflared exits with code 1 and logs ERR failed to unmarshal tunnel credentials. This happens when the tunnel was deleted and recreated on the Cloudflare dashboard outside of Burrow — the stored run token no longer matches the current tunnel. Fix: open Settings → Advanced → Delete tunnel to clear all state, then re-run the wizard. The wizard creates a fresh tunnel and stores the new run token in Keychain.

Port 22 not listening

Cause: cloudflared starts successfully but SSH connections time out at the local side. Verify the SSH daemon is running:

sudo launchctl list | grep ssh

On macOS, enable Remote Login in System Settings → General → Sharing → Remote Login. The tunnel ingress is fixed to ssh://localhost:22 — if sshd is not bound to that port, all connections fail locally.

Three failures in five minutes

Cause: CloudflaredManager counts exit events in a sliding 5-minute window. When the count reaches three, Burrow stops auto-restarting and posts a persistent notification. Fix: check the tunnel log category for the exit reason, resolve the underlying cause, then click Retry in the Burrow menu.

DNS issues

CNAME returns NXDOMAIN

Cause: the record was created with proxied: false, or manually deleted from the Cloudflare dashboard. A non-proxied CNAME to cfargotunnel.com always returns NXDOMAIN — Cloudflare does not expose an IP for tunnel targets without the orange-cloud proxy enabled. Fix: open the Cloudflare dashboard → DNS, find the CNAME for your subdomain, and set it to Proxied.

Subdomain already exists warning

Symptom: wizard step 5 shows "CNAME already exists at m4.hack.ski." Cause: a prior Burrow install or manual DNS entry left a record. Fix: choose a different subdomain, or delete the existing record from the Cloudflare dashboard before proceeding. Burrow will not overwrite an existing record automatically.

Tunnel does not recover after sleep

Cause: PowerObserver uses IORegisterForSystemPower to receive wake notifications. If the notification is missed on a rapid sleep/wake cycle, cloudflared remains stopped. Fix: click the Burrow menu icon and select Start tunnel for a manual restart. Check the lifecycle log category around the wake timestamp to confirm whether the notification was received.

Launch at login toggle has no effect

Cause: SMAppService.mainApp.register() fails silently when the app is not in /Applications. This is an Apple requirement for the modern login item API (Apple TN3127). Fix: quit Burrow, move Burrow.app to /Applications, relaunch, and toggle the setting again. Verify the item appears in System Settings → General → Login Items.

Cloudflare API returning 429

Cause: Cloudflare enforces 1200 requests per 5-minute window per account across all clients sharing the token. Burrow uses a single URLSession with no polling loops and applies exponential backoff on 429 automatically. If the error persists beyond a few minutes, check the cloudflare log category for repeated request URLs — a bug causing a request loop would appear there.

Persistent keychain errors

Cause: on first boot, the user login keychain may not be unlocked for 1–2 seconds. Burrow uses kSecAttrAccessibleAfterFirstUnlock and retries with backoff on errSecInteractionNotAllowed. If the error is persistent rather than transient, the app was likely moved or re-signed, which invalidates the keychain ACL. Fix: open Settings → Cloudflare → Change token and re-enter the token — this writes a fresh Keychain item under the current app signature.

Process.launch fails

Cause: App Sandbox is enabled. Burrow v1.0 ships with App Sandbox off so Process.launch() can reach cloudflared in /opt/homebrew/bin. If you built Burrow from source and accidentally enabled sandboxing, cloudflared will fail to launch with a permissions error. Fix: in Xcode → Signing & Capabilities, remove the App Sandbox entitlement and rebuild.
